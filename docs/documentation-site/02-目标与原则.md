# 02 - 目标与原则

## 目标

1. **单一文档站点** — 在 monorepo 根目录 `docs/` 维护一个 VitePress 实例，覆盖所有 `@wgl-m/*` 包
2. **统一入口** — 一个首页、一套 nav/sidebar，按包分栏（utils / plugins / node-utils / css）
3. **统一工具链** — VitePress、fast-glob 等 devDependencies 提升到根目录；删除 `packages/*/docs/package.json`
4. **统一部署** — 一次 `pnpm docs:build` 产出整站，CI 或单一 deploy 脚本推送 GitHub Pages
5. **规划与站点分离** — 内部方案文档通过 `srcExclude` 排除，不发布到对外站点

## 原则

| 原则 | 说明 |
|------|------|
| 内容优先搬迁 | 现有 `examples/*.md` 尽量原样迁移，减少重写 |
| 最小打断 | 迁移期可保留旧站点，新站上线后再下线 |
| 路径显式 | `@include` 改为从统一 docs 根出发的清晰相对路径 |
| 与 TS 迁移一致 | 文档引用 `.ts` 源码，不再引用已废弃的 `.js` / Webpack 产物路径 |
| 不加过度抽象 | 不引入自定义 VitePress 主题或复杂插件，沿用默认主题 |

## 不做的事

- 不把 VitePress 配置迁移到 TypeScript（保持 `.js` 或 `.mjs`，与 TS 迁移方案中「文档站不 TS 化」一致）
- 不在本阶段做全文 i18n（保持现有中文为主）
- 不迁移 monorepo 外的 `page__package-components` 文档（仅在 nav 保留外链）
- 不重写所有用例内容为 API Reference 自动生成（仍用手写 Markdown + `@include`）
- 不把 `RELEASE.md`、`MIGRATION.md` 全文搬进 VitePress（可选后续在「指南」栏增加摘要页）

## 验收标准

- [ ] 根目录 `pnpm docs` 可启动完整站点本地预览
- [ ] 根目录 `pnpm docs:build` 可构建且无 broken link
- [ ] utils、plugins 全部用例页可访问，`@include` 源码块正常渲染
- [ ] node-utils、css 至少有安装说明与索引页
- [ ] `typescript-migration/`、`documentation-site/` 不出现在站点 sitemap
- [ ] 各包 `package.json` 的 `homepage` 指向新统一 URL（或兼容旧 URL 的重定向页）
- [ ] 删除 `packages/*/docs` 下冗余的 VitePress 配置与 deploy 脚本
