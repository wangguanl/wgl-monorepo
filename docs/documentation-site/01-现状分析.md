# 01 - 现状分析

## 文档站点概览

| 包 | 文档目录 | VitePress | 部署目标 | homepage |
|----|----------|-----------|----------|----------|
| `@wgl-m/utils` | `packages/utils/docs/` | ✅ 1.6.3 | `page__package-utils` | wangguanl.github.io/page__package-utils/ |
| `@wgl-m/plugins` | `packages/plugins/docs/` | ✅ 1.6.3 | `page__package-plugins` | wangguanl.github.io/page__package-plugins/ |
| `@wgl-m/node-utils` | — | ❌ | — | — |
| `@wgl-m/css` | — | ❌ | 外部 `page__package-css` | wangguanl.github.io/page__package-css/ |

根目录 `docs/` 目前仅存放 **规划类 Markdown**（如 TypeScript 迁移方案），不参与任何 VitePress 构建。

## 现有架构

```
wgl-monorepo/
├── docs/
│   └── typescript-migration/     ← 规划文档，非站点
├── packages/
│   ├── utils/
│   │   ├── docs/                 ← 独立 VitePress 站点 A
│   │   │   ├── .vitepress/config.js
│   │   │   ├── examples/*.md     ← 含 @include 引用 ../../src/*.ts
│   │   │   └── deploy.sh         ← 推送到 page__package-utils
│   │   └── package.json          ← 依赖 vitepress、docs 脚本
│   └── plugins/
│       ├── docs/                 ← 独立 VitePress 站点 B（结构同上）
│       └── package.json
└── package.json                  ← docs:utils / docs:plugins 分别启动
```

`pnpm-workspace.yaml` 将 `packages/*/docs` 注册为独立 workspace 包，各站点各自维护 `package.json` 与 VitePress 缓存。

## 两份配置的重复点

两份 `.vitepress/config.js` 高度相似：

- 相同的 `lang`、`themeConfig` 文案（中文侧边栏、页脚、outline）
- 相同的 `fast-glob` 动态扫描 `examples/*.md` 生成 sidebar
- 相同的 `socialLinks`（GitHub monorepo）
- 仅在 `base`、`title`、nav 交叉链接 URL 上有差异

nav 中仍引用 **已迁出 monorepo 的旧仓库**（如 `page__package-components`），与当前 monorepo 结构不一致。

## 内容特征

- 文档以 **用例（examples）** 为主：安装说明 + API 示例 + `@include` 嵌入源码
- `@include` 路径相对各包 `docs/examples/`，形如 `../../src/debounce.ts`
- utils 文档还 `@include` 构建产物（`dist/es/*.mjs`），用于展示编译后代码
- 部分示例仍使用旧包名 `wgl-utils`（需随文档统一一并修正）

## 部署方式

各包 `deploy.sh` 流程相同：

1. `pnpm --filter @wgl-m/<pkg> docs:build`
2. 进入 `.vitepress/dist`，`git init` 后 force push 到独立 GitHub Pages 仓库

缺点：

- 无 CI 校验，依赖本地手动执行
- 两个站点 base path 不同（`/page__package-utils/` vs `/page__package-plugins/`），无法在同一域名下自然互链
- turbo `docs:build` 输出路径不统一（`docs/.vitepress/dist` vs 包内路径）

## 痛点

1. **维护成本高** — 双份 VitePress 配置、双份依赖、双份 deploy 脚本，改主题或导航要改两处
2. **体验割裂** — 用户需在两个 GitHub Pages 站点间跳转，无统一搜索与首页
3. **扩展困难** — `node-utils`、`css` 无文档入口；新增包需再复制一套 docs 脚手架
4. **与 monorepo 方向不一致** — TS 迁移、turbo 编排、changesets 发版已在根目录统一，文档仍分散在各包
5. **规划文档与站点文档混用 `docs/` 名称** — 根 `docs/` 是方案目录，包内 `docs/` 是 VitePress 根，新人易混淆

## 约束

- 已有 npm 包 `homepage` 字段指向旧 GitHub Pages URL，迁移需考虑链接兼容
- `@include` 依赖相对路径，搬迁后必须批量更新
- VitePress 1.6.3 与现有 markdown 特性（`::: code-group`、`layout: home`）需保持可用
- 规划类文档（`typescript-migration/`、`documentation-site/`）不应出现在对外站点导航中
